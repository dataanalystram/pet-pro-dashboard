## Table → Hook → Page Map

| Table | Hook | Pages (Consumer) | Notes |
|-------|------|-------------------|-------|
| `services` | `useServices()` | ServicesPage, DashboardPage, PublicBookingPage, PublicReviewPage, PublicPromoPage | Core entity. FK: reviews.service_id, service_staff.service_id, booking_requests.service_id |

| `orders` | `useOrders()` | OrdersPage, DashboardPage, ProductDetailPanel | Realtime enabled. FK target for: campaign_redemptions.order_id |
| `inventory` | `useInventory()` | InventoryPage, DashboardPage | Product catalog with stock tracking |

| `service_staff` | `useServiceStaff()` | ServicesPage, StaffPage, BookingRequestsPage | Junction: service ↔ staff with price_override |
| `staff_time_off` | `useStaffTimeOff()` | StaffPage (StaffTimeOffPanel) | Leave management with conflict detection |

## Data Flow: Pet Owner Integration Points

### Public-Facing Routes (Already Implemented)
| Route | Component | Description |
|-------|-----------|-------------|
| `/book/:serviceId` | PublicBookingPage | Pet owners submit booking requests |
| `/review/:serviceId` | PublicReviewPage | Pet owners leave reviews |
| `/promo` | PublicPromoPage | Pet owners browse active promotions |

### How Pet Owner Data Flows In

```
Pet Owner Action          →  Table Affected        →  Admin Sees
─────────────────────────────────────────────────────────────────
Submit booking request    →  booking_requests       →  BookingRequestsPage
Leave a review            →  reviews                →  ReviewsPage
Redeem promo code         →  campaign_redemptions   →  MarketingPage
Send a message            →  messages               →  MessagesPage
```

### Future Integration: Pet Owner Dashboard

When building a pet owner dashboard, these tables are the primary read targets:

1. **My Bookings**: `bookings` WHERE `customer_email = :email`
2. **My Requests**: `booking_requests` WHERE `customer_email = :email`
3. **My Orders**: `orders` WHERE `customer_email = :email`
4. **My Reviews**: `reviews` WHERE `customer_email = :email`
5. **My Messages**: `messages` WHERE `customer_email = :email`
6. **Available Services**: `services` WHERE `is_active = true`
7. **Active Promos**: `campaigns` WHERE `is_enabled = true AND status = 'active'`

> **Key Linking Field**: `customer_email` is the common identifier across
> bookings, booking_requests, orders, reviews, and messages. When adding
> auth for pet owners, create a `profiles` table linked to `auth.users`
> and match via email.

## Hooks Reference

### Generic CRUD (src/hooks/use-supabase-data.ts)
- `useSupabaseQuery<T>(table, options?)` — SELECT * with optional ordering
- `useInsert(table)` — INSERT with cache invalidation
- `useUpdate(table)` — UPDATE by id with cache invalidation  
- `useDelete(table)` — DELETE by id with cache invalidation

### Typed Queries (src/integrations/supabase/api.ts)
- `fetchServices()`, `fetchActiveServices()` — Service catalog
- `fetchBookingsByEmail(email)` — Customer's bookings
- `fetchRequestsByEmail(email)` — Customer's booking requests
- `fetchOrdersByEmail(email)` — Customer's orders
- `fetchReviewsByEmail(email)` — Customer's reviews
- `fetchMessagesByEmail(email)` — Customer's message thread
- `fetchActivePromos()` — Public-facing active campaigns
- `fetchStaffForService(serviceId)` — Available staff for booking
- `createBookingRequest(data)` — Submit new booking request
- `createReview(data)` — Submit new review
- `redeemPromoCode(code)` — Validate and apply promo

### Realtime (src/hooks/use-notifications.ts)
- `useNotifications()` — Auto-subscribes to realtime changes

## RLS Policy Notes
- All tables have RLS enabled
- Current policies allow public read (no auth required) for service provider use case
- **When adding pet owner auth**: Add policies scoped to `auth.uid()` matching customer profiles
- Use `customer_email` as the bridge between anonymous booking data and authenticated user profiles
